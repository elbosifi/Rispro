## 2. Current Theming Architecture

### Existing System
- **Design Tokens**: Complete CSS variable system defined in `index.css` (colors, shadows, radii, typography)
- **Design Language**: Industrial Skeuomorphism / Neumorphism
- **Styling Methods**: 3 concurrent approaches:
  1. CSS classes with CSS variables (V2 / Dashboard)
  2. Raw inline Tailwind classes (Legacy / Queue / Settings)
  3. Inline `style=` attributes (614 instances across codebase)
- **Tailwind Usage**: Only used for flexbox/grid layout - **not configured for theming**

### Strengths
- Well-defined CSS variable system with semantic tokens
- Consistent neumorphic shadow system
- Standardized button, card, and input classes exist
- Dark mode variables are present but inconsistently used

### Weaknesses
- No shared base components (Button, Card, Input)
- No Tailwind theme configuration (all colors only in CSS variables)
- Massive inline style explosion
- Two parallel design systems operating simultaneously

## 3. Inconsistencies Found

### Page Shell & Layout
- Page padding inconsistent: `p-4`, `p-6`, `px-8` across pages
- Header heights vary between 56px, 64px, and 72px
- Breadcrumb patterns differ between legacy and V2

### Card/Panel Consistency
| Aspect | Legacy | V2 |
|---|---|---|
| Base | `bg-white rounded-xl border shadow-sm` | `card-shell` with CSS variables |
| Radius | 12px fixed | 16px/32px via `--radius-xl` |
| Padding | `p-6` | `p-4` (1.5rem) |
| Borders | Explicit dark/light classes | Single `--border` variable |

### Buttons
- **Critical issue**: 2 completely separate button implementations
- Legacy: Inline Tailwind `bg-teal-600 hover:bg-teal-700`
- V2: Standard `btn-primary`, `btn-secondary`, `btn-ghost` classes
- Inconsistent padding: `py-3`, `py-2`, `0.65rem`
- Inconsistent heights: 48px, 44px, 40px

### Inputs/Selects/Textareas
- **Critical issue**: 5 distinct input styling implementations
- Heights: 6px (`py-1.5`), 8px (`py-2`), 10px (CSS class) vertical padding
- Legacy: Inline Tailwind with explicit colors
- V2: Standard `input-premium` class
- Settings: Ad-hoc inline styling
- DateInput: Separate custom implementation

### Tables
- No shared table component
- 3 different implementations across pages
- Cell padding varies: `p-2`, `p-3`, `py-2`
- Border widths: 1px, 2px
- Hover states inconsistent

### Dialogs/Modals
- No base modal component
- Every dialog implements own backdrop, positioning, styling
- 5 separate modal implementations exist

### Badges/Status Chips
- 4 different badge implementations
- V2 components use **hardcoded hex colors** breaking dark mode
- Queue page uses inline Tailwind
- Dashboard uses `pill-soft` CSS class

### Operational State Styling
| State | Legacy | V2 | Dashboard | Queue |
|---|---|---|---|---|
| Success | `text-teal-700` | `#15803d` | `var(--green)` | `text-emerald-700` |
| Warning | `text-amber-700` | `#a16207` | `var(--amber)` | `text-amber-700` |
| Error | `text-red-700` | `#991b1b` | `var(--accent)` | `text-red-700` |
| Disabled | `opacity-50` | `opacity: 0.6` | None | `disabled:opacity-30` |

### Loading/Empty/Error States
- No standardized patterns
- Each page implements own loading spinner
- Empty states have inconsistent wording and styling

## 4. Duplicated Patterns to Consolidate

1. **Button Component**: Merge 2 implementations into single shared component
2. **Card Component**: Create base Card component with standard variants
3. **Input Component**: Standardize all input fields to single implementation
4. **Badge Component**: Single StatusBadge with semantic variants
5. **Modal Component**: Base Dialog/Modal wrapper
6. **Table Component**: Standardized table with consistent styling
7. **Page Header**: Standard page title + action buttons pattern

## 5. Proposed Target Design-System Structure

### Layer 1: Tokens (No changes needed - already excellent)
- Keep existing CSS variables in `index.css`
- **Add Tailwind theme configuration** to expose tokens to Tailwind utilities
- No visual changes - just make existing tokens available to both systems

### Layer 2: Base Primitives (New shared components)
| Component | Variants | Purpose |
|---|---|---|
| `Button` | primary, secondary, ghost, icon | Single source of truth for all buttons |
| `Card` | default, elevated, compact | Standard card/panel wrapper |
| `Input` | default, textarea, select | Standardized input field with consistent height |
| `Badge` | success, warning, error, info, neutral | Status indicator chips |
| `Dialog` | base, confirm, alert | Modal wrapper component |

### Layer 3: Layout Primitives
- Standardize `PageContainer` to enforce consistent padding and max width
- Standardize page header pattern: Title + description + action buttons
- Add standardized Table wrapper component

### Layer 4: Domain Components
- Refactor existing domain components to use base primitives
- Remove inline styles and ad-hoc Tailwind classes

### State Styling Contract
Define ONE semantic mapping:
| State | Token |
|---|---|
| Available/Success | `--green` |
| Warning/Restricted | `--amber` |
| Error/Blocked | `--accent` |
| Info/Active | `--blue` |
| Disabled | `opacity-50` |
| Selected | `--accent` background with white text |

## 6. Execution Plan in Phases

### Phase 1: Foundation & Tokens (Safe, zero visual changes)
**Goal**: Make existing design system available to both CSS and Tailwind without changing any visuals

1. Add Tailwind theme configuration in `index.css` to map CSS variables to Tailwind utilities
2. Create shared base components directory structure
3. Build shared `Button` component with existing CSS classes (1:1 match)
4. Build shared `Card` component with existing `card-shell` class
5. Run full typecheck and test suite - zero visual changes expected

**Risk**: Low - only adding abstractions, no visual changes

### Phase 2: Normalize High-Impact Primitives
**Goal**: Replace duplicated implementations with shared components

1. Update all V2 appointment pages to use shared Button/Card
2. Update Dashboard to use shared components
3. Build shared `Input` component based on `input-premium`
4. Build shared `Badge` component with semantic variants
5. Replace hardcoded hex colors in V2 StatusBadge with CSS variables
6. Fix dark mode breakages

**Risk**: Medium - visual changes limited to consistency improvements

### Phase 3: Normalize Core Workflows
**Goal**: Bring legacy pages into alignment

1. Update Queue page to use shared components
2. Update Settings page to use shared components
3. Update Registration page to use shared components
4. Build shared Table component
5. Standardize page headers across all pages

**Risk**: Medium - legacy pages will visually align with V2 style

### Phase 4: Cleanup & Polish
**Goal**: Remove technical debt and inconsistencies

1. Remove remaining inline styles (target: <100 instances)
2. Build shared Dialog/Modal base component
3. Standardize loading/empty/error states
4. Audit and remove unused CSS classes
5. Document component usage patterns

**Risk**: Low - incremental cleanup

## 7. Risk Areas

1. **Breaking Existing Workflows**: All changes must be 1:1 functional replacements
2. **Dark Mode Breakages**: Hardcoded hex colors must be replaced with CSS variables
3. **Layout Shifts**: Changing input heights may cause minor layout shifts in forms
4. **Accessibility Changes**: Ensure focus states and keyboard navigation preserved
5. **Merge Conflicts**: Since this affects many files, changes should be small and incremental
